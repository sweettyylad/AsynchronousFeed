# Неоднозначности и принятые допущения

Каждый пункт: варианты трактовки → принятое решение → обоснование.
Ссылки на требования — по номерам из [REQUIREMENTS.md](./REQUIREMENTS.md).

---

## A1. Спаривание постов при разной длине списков (R3, R4)

**Неоднозначность.** Сторонний сервис может вернуть 10 постов для `cat` и 7 для `cat graffiti`. Task.md показывает только пример равных списков (`L1 R1 / L2 R2 / …`).

Варианты:
1. Обрезать по короткой стороне (`min`) — лента только из полных пар.
2. Дополнять по длинной стороне (`max`) — непарные позиции с `null` на одной стороне.

**Решение:** вариант 2 (`max`, недостающая сторона = `null`).

**Обоснование:** требование R8 уже обязывает отдавать ленту, где одна сторона целиком отсутствует (запрос ещё не завершён) — формат `{left, right|null}` в API нужен в любом случае. Прятать реально полученные посты (вариант 1) противоречит духу R8 «show the posts from the request that responded». UI отображает пустую ячейку/заглушку.

## A2. Что такое «current state of the requested feed» после перезагрузки (R6)

**Неоднозначность.** *«the user must be able to reload the page and … get the current state of the requested feed»* — где хранится знание о том, какой запрос «текущий»?

Варианты:
1. Бэкенд хранит «последний запрос пользователя» (нужны сессии/идентификация).
2. Текущий запрос кодируется в URL страницы (`/?q=cat`); бэкенд хранит результаты по query, фронтенд после F5 читает query из URL и запрашивает ленту.

**Решение:** вариант 2.

**Обоснование:** в Task.md нет ни авторизации, ни понятия пользователя как сущности — вводить сессии ради одного требования это оверинжиниринг (KISS). URL как носитель состояния — стандартный веб-паттерн (шареабельная ссылка — бонус). Ключевая часть требования — «without sending a request to the third-party service» — выполняется бэкендом: эндпоинт чтения ленты read-only и никогда не ходит в сторонний сервис.

## A3. Поведение read-эндпоинта при кэше старше 1 часа (R6 vs R9)

**Неоднозначность.** Пользователь перезагрузил страницу через 2 часа после запроса. Правило кэша (R9) говорит «старше часа — идём в сторонний сервис», правило перезагрузки (R6) — «без запроса к стороннему сервису».

Варианты:
1. GET ленты сам инициирует обновление, если данные протухли.
2. GET ленты всегда read-only, отдаёт то, что сохранено (даже протухшее); правило «1 час» применяется только при явном сабмите запроса (POST).

**Решение:** вариант 2.

**Обоснование:** R6 сформулировано безусловно («without sending a request to the third-party service»), а R9 описывает обработку именно *запроса пользователя* («if we have loaded posts for a query…, return them without making a request»). Разделение чисто: POST = «выполнить поиск» (применяет правило 1 часа), GET = «прочитать состояние» (никогда не ходит наружу). Побочный бонус: R6 выполняется даже если сторонний сервис лежит.

## A4. Гранулярность кэша: пара запросов или каждый upstream-запрос отдельно (R9)

**Неоднозначность.** *«posts for a query»* — «query» это то, что ввёл пользователь (`cat`), или каждый запрос к стороннему сервису (`cat`, `cat graffiti`)?

**Решение:** кэшируем каждый upstream-запрос отдельно; ключ — нормализованный текст (`trim`, схлопывание пробелов, `toLowerCase`).

**Обоснование:** максимальное переиспользование: если пользователь введёт `dog`, а затем `dog graffiti` как самостоятельный запрос, левая сторона второго поиска возьмётся из кэша правой стороны первого. Правило 1 часа при этом соблюдается для каждой стороны независимо, что согласуется с R8 (стороны и так живут независимо).

## A5. Нормализация query и лимиты ввода (I1)

**Неоднозначность.** Task.md не задаёт ограничений на текст запроса.

**Решение:** query — непустая строка после `trim`, максимум 100 символов, множественные пробелы схлопываются; регистр приводится к нижнему для ключа кэша. Пустая строка / только пробелы → 400.

**Обоснование:** разумные дефолты; лимит защищает БД и сторонний сервис от мусора. 100 символов достаточно для любого осмысленного поискового запроса.

## A6. Ретраи и трактовка rate limit (R10)

**Неоднозначность.** OpenAPI объявляет `429`, но не документирует лимиты и заголовок `Retry-After`.

**Решение:** на `429` и `5xx` — до 3 попыток с экспоненциальным backoff (1s → 2s → 4s) + jitter; уважать `Retry-After`, если сервис его пришлёт. После исчерпания попыток сторона получает статус `FAILED` с сообщением. Кэш (A4) и дедупликация одновременных одинаковых запросов — основная защита от лимитов.

**Обоснование:** стандартная практика при недокументированных лимитах; количество попыток и задержки — конфигурируемые константы.

## A7. Статус `FAILED` и повторная попытка

**Неоднозначность.** Task.md не описывает, что делать, если запрос к стороннему сервису окончательно упал.

**Решение:** сторона помечается `FAILED` (с сообщением об ошибке в API-ответе), UI показывает ошибку по этой стороне. Повторный сабмит того же query перезапускает загрузку упавшей стороны (FAILED не кэшируется по правилу 1 часа). «Зависшие» `PENDING` (процесс упал в момент фоновой загрузки) считаются протухшими через `PENDING_STALE_SECONDS` (120s) и тоже перезапускаются при следующем POST.

**Обоснование:** ошибки не должны залипать на час; самовосстановление через статусы в БД — простейшая альтернатива очереди задач.

## A8. Фронтенд: объём и стек

**Неоднозначность.** *«develop the frontend»*, но *«main focus should be on the backend»* — насколько богатым делать фронтенд?

**Решение:** одна страница: поле ввода + submit, лента пар (image + теги), индикаторы «сторона грузится» / «сторона упала», query в URL. Vite + TypeScript (strict), без UI-фреймворка. Собранная статика раздаётся самим NestJS.

**Обоснование:** фреймворк ради одной страницы с polling — лишняя зависимость и лишний вопрос на ревью. Один контейнер приложения упрощает compose и деплой (R12, R13).

## A9. API-токен стороннего сервиса

**Факт:** сервис требует заголовок `X-API-Token` (проверено: без токена и с произвольным токеном — `401 {"error":"Invalid or missing API token"}`). В Task.md токен не указан.

**Решение:** токен — переменная окружения `IMAGE_API_TOKEN`, обязательная (валидация при старте). Предполагаем, что токен выдаётся кандидату отдельно.

> **⚠️ Открытый вопрос к заказчику:** нужен рабочий `X-API-Token`. Без него нельзя проверить интеграцию вживую (разработка и тесты возможны — сторонний сервис в тестах мокается).

**ОТВЕТ:** Токен будет в .env в IMAGE_API_TOKEN

## A10. «Accessible via a link» — способ деплоя (R13)

**Решение:** деплой тем же docker-compose на дешёвый VPS / PaaS (Fly.io, Railway) — финальный шаг плана, не влияет на архитектуру. HTTPS — через платформу или reverse-proxy (Caddy) при необходимости.

**Обоснование:** Task.md не ограничивает способ; главное — работающая ссылка.
